# 核心概念

## LLM

LLM 是大脑，是今天大多数 AI 应用的能力底座。它能总结文档、写代码、生成计划、抽取结构化信息，也能把自然语言转成更适合程序处理的格式。但它本身不是数据库，不会自动知道外部世界的最新状态，也不能替系统承担事实校验。

我们要注意：

- 把模型当推理层，不当真相源：关键事实必须来自数据库、API、日志、文档源或其他可信系统。
- 把输出变成可检查对象：摘要、分类、计划和操作建议都应该尽量落成 schema、参数、引用或日志，而不是只停留在自然语言。
- 把不确定性前移暴露：模型不知道、资料过期、上下文不足时，要让系统显式降级，而不是编一个顺滑答案。

**基本工作原理： **

LLM 基于上下文以概率方式生成文本：给定一个输入，模型预测最可能的下一个词元序列。它们擅长语言理解、代码生成和推理，但在准确的事实回忆、确定性计算和跨会话状态维护方面表现不佳。

**四个控制层： **
上下文窗口是模型的“工作记忆”，控制模型在任何给定时间可以看到的信息量。系统指令设定身份、语气和行为边界。提示传达当前任务的意图。工具调用将模型从说话者转变为执行者。

**实际调用 LLM API： **
MaaS 允许您通过 API 密钥按令牌调用顶级模型，无需 GPU。核心参数： model 、 messages 、 temperature 、 max_tokens 。

**提示、工作流和代理之间的界限： **
提示是指让模型给出答案，由人做出决策。工作流是一个预定义的任务流程，其中模型是其中一个节点，路径是固定的。代理是指模型能够自主规划，动态调用工具，并在各个回合之间管理状态。这三者在故障模式、风险敞口和可调试性方面存在根本差异。

## Token

Token 是模型处理文本的基本单位。代码、JSON、长标识符、表格和混合语言文本都可以叫 token.

Token 直接影响三件事：上下文能放多少、调用成本是多少、模型能不能完整看见关键信息。长文档、代码、JSON、日志和多轮对话都很容易把上下文塞满。你需要决定哪些信息原样放入，哪些先压缩，哪些交给检索系统按需取回。

## Embedding

Embedding 是把文本、代码或其他对象映射成向量，用来衡量“语义上是否接近”。它常用于搜索、聚类、推荐、异常检测和 RAG。

Embedding 适合帮你从文档、知识库、代码仓库、讨论记录和产品日志中找相关材料。但它不适合单独判断“这个结论是否正确”。向量相似度只能说明内容接近，不能替代来源校验、规则检查和人工判断。

## Transformer

Transformer 是现代 LLM 的核心架构之一。它的关键能力来自 attention：模型可以在生成时关注输入中的不同位置，学习词、代码、事实和上下文之间的关系。

你不需要一开始就推公式，但要理解一个工程事实：Transformer 擅长在上下文里找模式，不等于它拥有稳定数据库。它能把文档、代码和用户目标组合起来生成解释，也可能因为上下文缺失或相似模式误导而给出错误归纳。

Transformer 给了模型强大的模式组合能力，但没有给它事实最终裁决权。

## Hallucination

Hallucination 指模型生成了看起来合理、但并不真实或无法验证的内容。它可能编造 API、错误解释代码、引用不存在的资料，或者把旧版本文档当成当前状态。

## Multimodal

Multimodal 模型可以处理文本、图片、音频、视频或屏幕截图。对 builders 来说，它的价值不是“更炫”，而是让模型读懂更多真实工作界面：图表、控制台、设计稿、应用页面、错误截图和确认弹窗。

## prompts

写给 AI 的「指令 / 提示词 / 问话」

说清身份（你是谁、让 AI 扮演谁）
说清任务（做什么）
说清格式 / 风格 / 要求

## Agent

Agent 是能围绕目标持续调用工具、读取状态、调整步骤的 AI 系统。它的关键不在“像人”，而在于它能不能在明确权限和可审计流程里，把建议推进到行动。

核心技术组件：

- 状态管理：多个节点共享对同一状态对象的读写访问权限。
- 长期记忆：跨会话存储和检索信息。
- MCP：LLM 与外部工具之间的统一连接协议。
- skills：支持自动发现和动态生成的可重用高级指令集。
- Tool calling：模型发出结构化请求；框架执行该请求并返回结果。
- 跟踪：可视化代理的执行链。
- 防护机制：输入/输出验证规则；如果违反规则，则执行停止。
- 交接：子任务完成后转移控制权。
- 错误恢复：失败时进行重试、回滚或升级到人工干预。

# AI 编码工具的价值与局限性：

Claude Code、Codex CLI 和 Cursor 可以快速生成样板代码、解释不熟悉的库并加速原型开发。然而，代码审查、测试设计和架构决策却无法委托给它们。

为什么人工智能的输出必须始终经过验证： 事实错误是最常见的陷阱：模型会自信地捏造信息，因此关键事实必须通过外部验证。模型在引用方面同样不可靠，它们经常会编造论文标题、URL 和数据来源，因此永远不要相信模型提供的任何链接。在更长的上下文中，推理漂移会成为一种风险，逻辑链会悄然断裂，结论偏离最初的前提，因此逐段验证至关重要。当涉及智能体时，还会出现两种额外的故障模式：执行越权，即智能体超出其授权范围执行操作（需要设置防护措施和人工干预检查点）；以及工具误用，即模型调用了错误的工具或传递了错误的参数（需要跟踪以监控执行过程）。

## 何时真正需要智能体？

当目标不明确、需要多个工具协作、中间结果决定下一步操作或状态需要在会话间保持时，可以使用智能体。如果是一次性问答（使用提示）、流程固定（使用脚本）、合规性要求严格（使用人工审核节点）或数据确定性至关重要（使用数据库查询），则应坚持使用更简单的解决方案。复杂性和风险越高，就越应该谨慎避免过度使用智能体。

# 框架

## LangChain

LangChain 是最常见的 LLM 应用开发框架之一，覆盖模型接入、prompt、工具调用、retriever、agent、output parser 等模块。

它适合快速把模型能力和外部系统接起来，也适合学习 AI 应用常见组件。但使用时要小心“抽象太早”：如果你还没搞清楚自己的工作流，直接套一堆 chain 和 agent，后面排查问题会变难。

LangChain 更像一套组件库。它适合帮你组合能力，不适合替你决定产品边界。

## LangGraph

LangGraph 更偏向工作流和状态机。它把 Agent 或多步骤任务表示成 graph：节点负责执行动作，边负责控制流，state 负责记录过程。

当任务只是“问一次，答一次”，LangGraph 可能太重。
但如果任务需要多轮工具调用、重试、人工确认、分支、恢复和长期运行，显式 graph 会比一长串 prompt 历史更可靠。

一个实用判断是：只要你开始关心任务走到哪一步、能否恢复、失败后从哪里继续，就应该考虑 graph / state machine。

## OpenAI Agents SDK

OpenAI Agents SDK 用来构建带工具、handoff、guardrails、tracing 的 Agent 应用。它的价值在于把 Agent 工作流里的常见工程问题做成可组合结构。

你可以用它组织：

- Agent 的指令和工具
- 多 Agent 之间的 handoff
- 工具调用和结构化输出
- guardrails 和运行时追踪
  关键仍然是边界：SDK 可以帮你执行流程，但你要定义哪些工具可用、哪些动作需要确认、什么输出算失败。

## DSPy

DSPy 关注的是把 prompt / LM pipeline 写成可优化的程序。它不是让你手工调一句 prompt，而是把输入输出、模块和指标定义清楚，再用 optimizer 改进 prompt 或调用策略。

当你的任务只是写一段文案，DSPy 未必必要。
当你有明确数据集、评估指标和可重复任务时，它会更有价值，比如分类、抽取、问答、rerank、复杂推理链。

DSPy 的关键启发是：不要只靠感觉调 prompt，要让任务、数据和指标进入系统。

## Hermes

Hermes 在这里更适合作为“面向工具调用和结构化输出的模型 / agent 生态”来理解，而不是通用框架。

它提醒我们一件事：框架不是唯一抽象层。模型本身是否擅长 tool calling、JSON mode、long context、reasoning，也会影响系统设计。一个工具调用能力稳定的模型，可以减少很多解析和兜底成本；反过来，模型能力不稳定时，再好的框架也要补大量 guard。

看 Hermes 这类方向时，不要只看榜单分数，要看它是否能稳定地产生你需要的结构化输出和工具调用格式。

## Learning Agent

Learning Agent 指系统可以从反馈、日志、评估结果或用户修正中改进。这里的“学习”不一定是训练模型，也可以是更新 prompt、调整 retriever、补充规则、改进测试集。

真实产品里，Learning Agent 最容易踩的坑是把线上反馈直接变成行为变化。这样会引入数据污染、越权学习和不可解释变化。

更稳的流程是：

记录失败样本。
人工或规则标注原因。
加入 eval / regression set。
修改 prompt、检索、工具或模型配置。
通过测试后再上线。
学习能力要先进评估闭环，再进入生产系统。

# 评估

Evaluation 是把“感觉效果不错”变成“系统可持续改进”的方法。没有 eval，prompt、模型、RAG、Agent 和工具调用的变化都只能靠主观试用判断，迟早会被回归问题拖住。

用一套标准，量化打分 / 判定好坏，检验大模型、提示词、智能体、API 输出结果够不够准、稳、合规、好用。

## 核心目的

- 测出模型回答正确率
- 对比多个模型谁更强
- 排查幻觉、胡说、违规、答非所问
- 验收 Prompt、Agent 流程是否合格
- 上线前把关、上线后监控质量

## 分类

- 人工评估：人肉眼打分
- 自动评估：程序 / AI 自动打分
- 线下数据集测评：用标准题库批量跑分

## 评估指标

- 准确率：答案是否正确
- 相关性：有没有跑偏、答非所问
- 事实一致性：有无编造信息（防幻觉）
- 流畅度：语句通顺自然
- 合规性：是否涉黄暴、敏感内容
- 召回 / 精准率（知识库 RAG 专用）：有没有搜到有用内容
- 耗时、token 消耗、成本：性能 & 成本评估
