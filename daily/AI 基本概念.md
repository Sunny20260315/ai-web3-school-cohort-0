# 核心概念

## LLM

LLM 是大脑，是今天大多数 AI 应用的能力底座。它能总结文档、写代码、生成计划、抽取结构化信息，也能把自然语言转成更适合程序处理的格式。但它本身不是数据库，不会自动知道外部世界的最新状态，也不能替系统承担事实校验。

我们要注意：

- 把模型当推理层，不当真相源：关键事实必须来自数据库、API、日志、文档源或其他可信系统。
- 把输出变成可检查对象：摘要、分类、计划和操作建议都应该尽量落成 schema、参数、引用或日志，而不是只停留在自然语言。
- 把不确定性前移暴露：模型不知道、资料过期、上下文不足时，要让系统显式降级，而不是编一个顺滑答案。

**基本工作原理： **

LLM 基于上下文以概率方式生成文本：给定一个输入，模型预测最可能的下一个词元序列。它们擅长语言理解、代码生成和推理，但在准确的事实回忆、确定性计算和跨会话状态维护方面表现不佳。

**四个控制层： **
上下文窗口是模型的“工作记忆”，控制模型在任何给定时间可以看到的信息量。系统指令设定身份、语气和行为边界。提示传达当前任务的意图。工具调用将模型从说话者转变为执行者。

**实际调用 LLM API： **
MaaS 允许您通过 API 密钥按令牌调用顶级模型，无需 GPU。核心参数： model 、 messages 、 temperature 、 max_tokens 。

**提示、工作流和代理之间的界限： **
提示是指让模型给出答案，由人做出决策。工作流是一个预定义的任务流程，其中模型是其中一个节点，路径是固定的。代理是指模型能够自主规划，动态调用工具，并在各个回合之间管理状态。这三者在故障模式、风险敞口和可调试性方面存在根本差异。

## Token（词元）

### 最简单的解释

Token 是模型理解语言的最小单位，类似于"字"，但不完全是。

### 具体例子

```
"Hello"           → 1 token
"world"           → 1 token
"!"               → 1 token
"你好"            → 1 token（多语言优化）
"2024"            → 1 token

代码示例：
function() {}     → 4 tokens
var x = 123;      → 5 tokens
```

💡 **关键点**：中文一般比英文消耗更多 tokens（中文通常 1 个字 = 1 token，英文 1 个词 = 1～2 tokens）

### Token 对你的三大影响

1. **成本** - 费用直接按 token 计算
   - 100 个 tokens ≈ $0.003（Claude 3 Sonnet）
   - 长对话 = 高成本

2. **速度** - Token 越多，模型响应越慢
   - 100 tokens ≈ 几毫秒
   - 100K tokens ≈ 几秒钟

3. **准确性** - 上下文太长，模型容易"遗忘"重点
   - 短提示（< 1K tokens）：模型专注力强
   - 长提示（> 50K tokens）：容易分散注意力

### 常见误解

❌ **误解**："1 token = 1 个字"

- 标点、空格、缩进都算 token
- 特殊符号可能占多个 token
- 修正：用 tokenizer 工具估算，不要凭感觉

### 实战技巧：如何减少 token 消耗

**不好的做法**（浪费 token）：

```
提示："你好，今天天气真好。我想问你一个关于 Python 的问题。
在 Python 中，我想写一个函数，这个函数的目的是...
详细要求是...
另外还要..."
```

预计：500+ tokens（很多是废话）

**好的做法**（节省 token）：

```
问题：写 Python 函数检查 email 格式
要求：
- 返回 True/False
- 检查 @、. 符号
```

预计：50 tokens（清晰、简洁、成本低）

### Token 管理策略

| 情况       | 方案                                | 效果      |
| ---------- | ----------------------------------- | --------- |
| 长文档分析 | 先压缩再分析，或用检索提取关键部分  | 成本↓ 70% |
| 多轮对话   | 定期清空历史，只保留最近几条        | 成本↓ 50% |
| 代码审查   | 用代码摘要替代全量代码              | 成本↓ 80% |
| 大数据集   | 用 embedding + 检索，而不是直接放入 | 成本↓ 90% |

## Embedding（向量化）

### 最简单的解释

Embedding 就是把文字转成一堆数字，这堆数字可以用来衡量两段文字有多"接近"。

```
"猫是动物" → [0.23, -0.45, 0.89, ... 1536 个数字]
"狗是动物" → [0.24, -0.43, 0.91, ... 1536 个数字]
"月球在太空" → [0.01, 0.05, 0.12, ... 1536 个数字]

计算相似度：
猫 vs 狗：相似度 0.95（很接近）
猫 vs 月球：相似度 0.12（完全不同）
```

### 实战应用

**1. 智能搜索（最常用）**

```
场景：用户问 "如何处理焦虑"
系统流程：
1. 把用户问题转成向量
2. 在知识库中搜索最相似的文档
3. 返回最相关的文章（不是关键词匹配）
结果：精准率高 ✅
```

**2. 推荐系统**

```
场景："你可能喜欢这个文章"
原理：找和你读过文章最相似的其他文章
```

**3. 异常检测**

```
场景：检测网络流量中的异常
原理：异常的数据向量离"正常向量群"很远
```

**4. RAG（检索增强生成）**

```
场景：用自己的知识库回答问题
流程：
1. 用户问题 → 转成向量
2. 在知识库中搜索相似文档
3. 把相关文档当作上下文给 LLM
4. LLM 基于这个上下文回答
结果：准确、有依据
```

### 常见误解

❌ **误解 1**："Embedding 相似 = 内容正确"

- 两段文字可能在语义上很接近，但一个是真的，一个是假的
- 例："月球是用芝士做的" vs "月球的成分"
  - 相似度可能很高（都在讲月球成分）
  - 但前者是错的！
- 修正：embedding 用来找"相关内容"，不是找"正确内容"

❌ **误解 2**："Embedding 可以替代事实核实"

- 向量接近 ≠ 事实核实
- 必须配合其他方法（API 查询、数据库检查、人工审核）

❌ **误解 3**："所有 embedding 模型效果一样"

- 不同模型性能差异很大
- 中文效果：OpenAI < 国产模型
- 代码效果：代码专用 embedding > 通用 embedding

### Embedding 的使用边界

| 用途       | 适合 ✅    | 不适合 ❌              |
| ---------- | ---------- | ---------------------- |
| 相似度搜索 | 文档、代码 | 结构化数据（用数据库） |
| 推荐系统   | 内容推荐   | 实时库存推荐           |
| 异常检测   | 通用日志   | 数值计算               |
| 分类       | 文本分类   | 严格业务规则分类       |

## Transformer（模型架构）

### 最简单的解释

Transformer 是一种"神经网络"架构，是现代 AI 模型的大脑。不需要你理解细节，只需要知道：

**它能做什么：**

- 理解整句话的含义（不是逐字处理）
- 找出哪些词之间最相关（Attention 机制）
- 生成符合逻辑的后续内容

**简单类比：**

```
普通模型："我爱吃"
         → 一个词一个词处理
         → 可能得到："我爱吃苹果...水果...甜甜的..."
         → 前后可能不连贯

Transformer："我爱吃"
           → 同时看整句话
           → "爱"和"吃"关系强 → 理解是表达喜欢
           → 生成合理后续："我爱吃苹果，尤其是红苹果"
           → 前后连贯、有逻辑
```

### Attention（注意力）机制

**原理（大白话）：**

- 当模型生成下一个词时，它可以"回头看"输入中的每个词
- 根据与下一个词的关系程度，给每个词分配"权重"
- 利用这些权重生成最合适的词

**例子：**

```
输入："The bank is on the river"

当生成"bank"附近的词时：
- "bank"与"river"关系强（都是地点）→ 权重高
- "bank"与"the"关系弱 → 权重低
- 结果：生成与"地点"相关的后续词（如"building"、"financial"等）
```

### 关键理解：Transformer 的能力边界

**✅ 擅长**

- 找到上下文中的模式
- 理解词与词之间的关系
- 生成符合规律的文本
- 写代码、总结文档、翻译

**❌ 不擅长**

- 记住"绝对事实"（如"2024 年的 ETH 价格"）
- 进行精确数值计算
- 维护跨会话的持久记忆
- 访问实时信息

### 常见误解

❌ **误解 1**："Transformer 模型知道一切"

- 模型知识 = 训练数据（通常是半年～2 年前的数据）
- 实时信息：通过工具调用、API 查询获得

❌ **误解 2**："Transformer 的权重就是"理解"

- 权重是数学参数，不是真正的"理解"
- 可能是巧合的模式匹配，不是真正的因果关系

### 工程启示

Transformer 的强大意味着：

- 我们可以让模型处理长上下文（像整个代码库）
- 但这也意味着上下文很重要 → **提示词质量决定输出质量**
- 必须配合其他系统（数据库、API、验证）来弥补不足

## Hallucination（幻觉/编造）

### 最简单的解释

Hallucination 就是 AI 模型"胡编乱造"——看起来很有信心，但其实在编假信息。

### 常见的幻觉类型

| 类型             | 例子                                            | 后果                 |
| ---------------- | ----------------------------------------------- | -------------------- |
| **编造事实**     | "2024 年比特币价格是 $50,000"（错的）           | 用户做错决策         |
| **编造链接**     | "详见 https://example.com/article123"（不存在） | 用户点击打不开       |
| **编造代码函数** | "使用 `numpy.fast_matrix()`（不存在的函数）"    | 代码运行崩溃         |
| **编造论文**     | "根据 Smith 2023 的研究..."（这篇论文不存在）   | 引用无效，学术不诚实 |
| **过时信息**     | "TensorFlow 最新版本是 2.5"（现在是 2.13）      | 代码不兼容           |
| **自相矛盾**     | 前一句说"A 是真的"，后一句说"A 是假的"          | 用户困惑             |

### 幻觉发生的原因

❓ **问题 1**：为什么模型会自信地编造？

- 模型的工作方式是"预测最可能的下一个词"
- 如果训练数据中有类似的模式，模型会"补全"
- 例：训练数据中有很多"论文标题：....."这样的格式，所以模型会生成看起来像论文标题的假标题

❓ **问题 2**：为什么不能直接相信模型的自信度？

- 模型的"自信度"与准确性无关
- 编造的信息和真实信息一样有信心
- 就像一个非常自信的骗子

### 实战：如何防止和检测幻觉

**防止幻觉**

```
❌ 差做法（诱发幻觉）：
用户问："iOS 18 的新功能是什么？"
模型："iOS 18 有以下新功能：1. ... 2. ... 3. ..."
风险：可能编造功能，尤其是如果 iOS 18 还没发布

✅ 好做法（防止幻觉）：
1. 先告诉模型它的知识截止时间
2. 给出权威来源（官方文档 URL）
3. 让模型先说"我不确定"而不是编造
   系统指令："如果你不确定，直接说 '我不知道'，不要编造"
4. 让模型附带来源

提示词示例：
"根据我给你的官方文档，iOS 18 的新功能有哪些？
如果文档中没有提到某个功能，不要编造，直接说'文档中没有提到'。"
```

**检测幻觉**

- 检查所有链接是否真实存在
- 验证引用的数据/论文是否真的存在
- 让模型提供具体来源而不是笼统的"根据我的知识"
- 用独立搜索验证关键事实

## Prompts（提示词）

### 最简单的解释

Prompt = 写给 AI 的"指令"。写得好的 Prompt 能让 AI 输出质量提升 50%～200%。

### Prompt 三大要素

| 要素          | 作用           | 例子                                         |
| ------------- | -------------- | -------------------------------------------- |
| **身份**      | 告诉 AI 你是谁 | "你是一个有 10 年经验的 Python 工程师"       |
| **任务**      | 说清楚做什么   | "帮我分析这段代码的性能问题"                 |
| **格式/要求** | 定义输出形式   | "用 JSON 格式返回，包含问题、原因、解决方案" |

### Prompt 的好坏对比

❌ **差的 Prompt**（模糊、模糊、还是模糊）

```
"帮我写代码"
结果：模型不知道要写什么、用什么语言、什么规范

"分析这篇文章"
结果：模型不知道从哪个角度分析、深度是什么

"回答这个问题"
结果：无法预测输出质量和格式
```

✅ **好的 Prompt**（清晰、有例子、有约束）

```
"你是一个经验丰富的 Python 代码审查员。
请审查以下代码，指出：
1. 潜在的性能问题
2. 内存泄漏风险
3. 代码可读性改进
4. 安全问题
返回格式：Markdown，用【问题】【原因】【修复】的三级标题组织。
代码如下：
[代码]"

结果：
- 格式一致 ✅
- 内容完整 ✅
- 易于解析 ✅
- 可复现 ✅
```

### 常见误解与解决方案

❌ **误解 1**："Prompt 就是随便说说"

- Prompt 质量直接决定输出质量
- 同一个需求，好 Prompt vs 差 Prompt 的输出质量天差地别
- 修正：投入时间精心设计 Prompt

❌ **误解 2**："越长的 Prompt 越好"

- 长 Prompt = 更多 token = 更高成本 + 模型容易分散注意力
- 关键：清晰 > 长
- 修正：用最少的词说清楚需求

❌ **误解 3**："Prompt 对所有模型都一样"

- 不同模型对 Prompt 的理解能力不同
- Claude 理解复杂 Prompt 能力强
- GPT 可能更适合结构化 Prompt
- 修正：根据模型调整 Prompt

### Prompt 优化技巧

**技巧 1：例子的力量（Few-shot Prompting）**

```
不好：
"把英文翻译成中文"

好：
"把英文翻译成中文，用户友好的自然表达。

例子：
- 'Play video' → '播放视频'
- 'Settings' → '设置'
- 'Dark mode' → '深色模式'

现在翻译：'Advanced features'"
```

**技巧 2：分步骤思考**

```
不好：
"总结这篇文章"

好：
"按以下步骤总结文章：
1. 读通文章，找出主要观点
2. 列出 3 个关键事实
3. 用一句话总结核心论点
4. 指出可能的局限性

文章内容：[...]"
```

**技巧 3：约束输出格式**

```
不好：
"分析代码的问题"

好：
"分析代码，返回 JSON：
{
  'issues': [
    {
      'type': '性能',
      'severity': 'high|medium|low',
      'description': '...',
      'fix': '...'
    }
  ]
}"
```

## Agent（智能体）

### 最简单的解释

Agent = AI + 自主权 + 工具

一个 Agent 可以：

1. **理解目标** - 明白要做什么
2. **自主规划** - 自己决定用什么步骤
3. **调用工具** - 执行具体操作（调用 API、查询数据库、写文件）
4. **动态调整** - 根据中间结果改变策略
5. **记住上下文** - 跨多步骤记住重要信息

**与 Prompt 的区别**

```
Prompt（一问一答）：
用户 → 问题 → LLM → 回答 → 结束
风险：低，LLM 只是回答

Agent（多步自主）：
用户 → 目标 → LLM → 检查→调用工具→读结果→调整方案→执行→再次检查→...
风险：高，需要防护机制
```

### 实战例子：何时需要 Agent？

**例子 1：简单任务（用 Prompt 就够）**

```
用户问："什么是区块链？"
解决方案：Prompt
为什么：一问一答，不需要工具，不需要中间步骤
```

**例子 2：中等任务（可能需要 Agent）**

```
用户问："帮我查一下 Apple 股票价格，然后告诉我和去年同期比是升是跌"
流程：
1. LLM 决定：需要查股票价格 → 调用 Price API
2. 得到结果：$150（现在），$120（去年）
3. LLM 比较 → 生成答案："上升了 25%"
解决方案：Agent（需要工具调用）
为什么：需要实时数据，LLM 自己做不了
```

**例子 3：复杂任务（一定需要 Agent）**

```
用户需求："自动监控我的 GitHub repo，当发现新的 issue 时：
- 先自动分析问题
- 如果是 bug，自动生成一个修复 PR
- 如果是功能请求，自动生成一个需求分析文档
- 最后发送消息给我"

流程：
1. Agent 监听 GitHub webhooks
2. 新 issue 出现 → 获取内容
3. 分析是什么类型 → 调用分类工具
4. Bug？ → 调用代码分析 + 自动修复工具 → 生成 PR
5. 功能请求？ → 调用文档生成工具
6. 调用通知工具 → 发消息
7. 记录这次执行 → 下次参考

解决方案：Agent（多步骤、多工具、跨系统协调）
为什么：需要持续监听、中间决策、多工具配合、状态记忆
```

### Agent 的关键风险

**风险 1：执行越权**

```
场景：
- 给 Agent 权限：可以查看所有代码库
- 用户问："帮我把所有 TODO 注释删掉"
- Agent 不理解"所有"是什么意思，删掉了公司所有项目的 TODO
- 结果：大量代码失去注释 ❌

防护：
- 限制 Agent 的权限范围（只能访问特定仓库）
- 要求人工审核危险操作（删除、修改）
- 所有操作前显示"确认"弹窗
```

**风险 2：工具误用**

```
场景：
- Agent 想查数据库中的用户信息
- 但选错了工具，用了生产数据库而不是测试数据库
- 结果：真实用户数据被 Agent 随意操作 ❌

防护：
- 明确标记每个工具的用途和风险等级
- Agent 选择工具前需要"确认工具"
- 高风险工具（删除、修改真实数据）需要人工批准
```

**风险 3：信息遗忘导致错误**

```
场景：
- Agent Step 1：查询"我的预算是 $1000"
- Agent Step 5：忘了这是 $1000 → 花了 $5000 ❌

防护：
- 使用显式状态管理（LangGraph、Agent SDK）
- 每步都记录关键信息
- 定期（每 3～5 步）总结已知信息
```

### 常见误解

❌ **误解 1**："Agent 就是自动化"

- Agent ≠ 完全自动
- 很多情况下需要"人在回路"（Human-in-the-Loop）
- 高风险操作必须人工审核

❌ **误解 2**："Agent 越聪明越好"

- 越聪明 = 越可能做超出预期的事
- 简单、可预测的 Agent 比复杂 Agent 更安全
- 修正：选择"够用但可控"的 Agent

❌ **误解 3**："一个通用 Agent 可以做一切"

- 最好的 Agent 是专门设计的
- 针对特定任务的 Agent 效率高、出错少
- 修正：为不同任务设计不同 Agent

### 何时真正需要 Agent？

**✅ 用 Agent**

- 目标复杂、需要多个步骤
- 中间结果决定下一步操作（分支逻辑）
- 需要调用多个工具
- 需要跨会话的持久记忆
- 可以容忍一定风险

**✅ 不用 Agent，用 Prompt**

- 一次性问答（"这是什么？"）
- 流程简单固定（写一篇文章）
- 没有工具调用需求
- 高风险场景（金融交易、医学诊断）

**✅ 不用 Agent，用脚本**

- 完全确定性的流程
- 需要精确控制每一步
- 性能要求高
- 维护性要求高

### Agent 的架构模式

| 复杂度   | 方案               | 适用场景           | 风险             |
| -------- | ------------------ | ------------------ | ---------------- |
| **低**   | Prompt + 函数      | 简单工具调用       | 低               |
| **中**   | Prompt + LangChain | 多工具、基础流程   | 中               |
| **高**   | LangGraph + State  | 复杂流程、状态管理 | 可控             |
| **最高** | 多 Agent + 编排    | 跨系统协调         | 高（需严格防护） |

# AI 编码工具的价值与局限性

Claude Code、Codex CLI 和 Cursor 是强大的编程助手，但它们的能力有明确的边界。

### AI 编码工具能做什么 ✅

| 任务             | 例子                      | 精准度  |
| ---------------- | ------------------------- | ------- |
| **生成样板代码** | 创建项目框架、API 端点    | 高 ✅   |
| **代码补全**     | 根据前面的代码猜测下一行  | 高 ✅   |
| **Bug 修复建议** | "这里可能出错，原因是..." | 中等 ⚠️ |
| **库/API 讲解**  | "这个函数的用法是..."     | 高 ✅   |
| **重构建议**     | "这段代码可以简化为..."   | 中等 ⚠️ |
| **单测生成**     | 根据代码自动生成测试用例  | 中等 ⚠️ |

### AI 编码工具做不好的事 ❌

| 任务         | 为什么                     | 后果           |
| ------------ | -------------------------- | -------------- |
| **代码审查** | 容易遗漏设计问题、安全漏洞 | 问题进入生产   |
| **架构决策** | 缺乏业务上下文             | 设计不匹配需求 |
| **性能优化** | 不知道真实数据量、瓶颈     | 优化无效或反向 |
| **安全审计** | 容易忽略边界情况           | 安全漏洞       |
| **验收测试** | 不理解用户的真实期望       | 测试用例不完整 |
| **技术选型** | 缺乏生产经验、成本考量     | 选错技术栈     |

### 实战：如何正确使用 AI 编码工具

**❌ 错误做法（信任 AI 的所有输出）**

```
流程：
1. "帮我写一个登录功能"
2. Claude Code 生成代码
3. 直接部署到生产
4. 用户反馈：密码明文存储 💀

为什么出错：
- AI 快速生成代码，但没考虑安全
- 没有人工审核
```

**✅ 正确做法（AI 辅助，人工把关）**

```
流程：
1. "帮我写一个登录功能，使用 bcrypt 哈希密码"
2. Claude Code 生成代码
3. 人工代码审查：
   - 检查密码哈希是否正确
   - 验证 SQL 注入防护
   - 检查会话管理
4. 运行单测 + 集成测试
5. 安全审计
6. 部署到生产

关键：AI 是生成工具，人是把关者
```

### Multimodal（多模态）

**Multimodal 是什么**
多模态模型可以处理：文本 + 图片 + 代码 + 视频等多种格式

**对开发者的实际价值**

| 场景                            | 用处             |
| ------------------------------- | ---------------- |
| 看错误截图，AI 直接诊断问题     | 加速 debug       |
| 上传设计稿，AI 生成 HTML/CSS    | 快速原型         |
| 上传控制台日志截图，AI 分析问题 | 快速诊断         |
| 上传代码报错截图，AI 解释       | 省去手工复制粘贴 |

**不要过度期待**

- AI 看截图 ≠ AI 理解设计意图
- 设计和实现之间还是需要人做决策

# 框架

## LangChain

### 最简单的解释

LangChain 是 LLM 应用的"乐高积木"。它提供了一套现成的组件（模型接入、Prompt、工具调用、RAG 等），让你快速组合出完整应用。

### 什么时候用 LangChain ✅

| 场景           | 理由                       |
| -------------- | -------------------------- |
| **快速原型**   | 框架加速，减少低层 bug     |
| **多工具集成** | 有现成的连接器，省时间     |
| **RAG 系统**   | 有 vector store 集成，够用 |
| **学习**       | 文档全、社区大、示例多     |

### 什么时候不用 LangChain ❌

| 场景           | 理由                   |
| -------------- | ---------------------- |
| **高性能**     | 抽象层多，开销大       |
| **复杂工作流** | 用 LangGraph，别套娃   |
| **完全定制**   | 限制多，不如直接调 API |
| **成本敏感**   | 默认配置浪费 token     |

### 常见误解

❌ **误解 1**：LangChain 可以解决所有问题

- 只是工具，不是解决方案
- 修正：关键设计还是你做

❌ **误解 2**：LangChain 代码一定比手写 API 好

- 可能更复杂、更慢
- 修正：简单任务别过度设计

## LangGraph

### 最简单的解释

LangGraph 是把复杂工作流画成图的框架。每个任务是一个"节点"，流程是"边"，整个过程中的信息是"状态"。

### LangGraph vs LangChain

| 对比     | LangChain | LangGraph          | 选择         |
| -------- | --------- | ------------------ | ------------ |
| 学习曲线 | 简单      | 陡                 | 简单选 Chain |
| 适合任务 | 线性流程  | 分支、循环、多工具 | 复杂选 Graph |
| 调试     | 中等      | 容易               | Graph 更容易 |
| 性能     | 还行      | 更快               | Graph 性能好 |

### 什么时候用 LangGraph ✅

复杂 Agent 自动修复流程（多分支、循环、需要人工确认）→ 用 LangGraph

### 什么时候不用 LangGraph ❌

简单翻译（输入→处理→输出）→ 直接调 API 或 LangChain

### 常见误解

❌ **误解 1**：LangGraph 比 LangChain 厉害，应该用 Graph

- 只是表达方式不同，不是性能优越
- 修正：简单用 Chain，复杂用 Graph

❌ **误解 2**：LangGraph 会自动处理一切

- 只是框架，错误处理要自己写
- 修正：Graph 是工具，不是自动化

### 判断是否用 Graph

满足任一即用 Graph：

- [ ] 有分支逻辑（if/else）
- [ ] 有循环（重试、反复调整）
- [ ] 需要人工干预
- [ ] 需要跨步骤记忆信息
- [ ] 任务失败需要从中途恢复

## OpenAI Agents SDK

### 最简单的解释

OpenAI Agents SDK 是 OpenAI 官方的 Agent 框架，重点是"安全可控"——定义工具权限、执行规则、审计日志。

### 核心特性

| 特性              | 说明                | 用处          |
| ----------------- | ------------------- | ------------- |
| Handoff           | Agent 之间交接任务  | 多 Agent 系统 |
| Guardrails        | 定义 Agent 能做什么 | 防止越权      |
| Structured Output | 强制输出格式        | 可靠解析      |
| Tracing           | 完整执行日志        | 调试+审计     |

### 什么时候用 ✅

- 需要多个 Agent 协作
- 有复杂权限控制
- 生产系统需要审计日志
- 需要结构化输出（JSON）

### 常见误解

❌ **误解**：Guardrails 能完全防止坏事

- Guardrails 只是约束，聪明 Agent 可能绕过
- 修正：Guardrails + 人工审核才是真防护

### 实战建议

错误做法 - 给 Agent 过多权限：给 delete_user_tool、create_admin_tool 这样危险工具

正确做法 - 分角色、分权限：admin_agent 有管理权限，user_agent 只读

## DSPy

### 最简单的解释

DSPy 把 LLM 应用写成"程序"，有输入、输出、验证、优化。不是手动调 Prompt，而是自动优化 Prompt。

### 手动调 Prompt 的问题

反复调整 → 没有量化指标 → 不知道是否已经最优

### 用 DSPy 的方式

1. 定义输入、输出、验证标准
2. 提供训练数据
3. 自动优化 Prompt（不用手动调）
4. 生成最优版本

### 什么时候用 ✅

- 有明确数据集（100+ 样本）
- 有衡量标准（准确率、相关度等）
- 任务重复运行（会不断优化）
- 在意成本（优化能减少 token）

### 什么时候不用 ❌

- 一次性任务
- 没有量化标准
- 数据集太小（<10 样本）

### 常见误解

❌ **误解 1**：DSPy 会自动把烂 Prompt 变好

- DSPy 基于数据优化，数据烂→优化也没用
- 修正：DSPy 需要高质量训练集

❌ **误解 2**：DSPy 替代了人工微调

- 只是自动调 Prompt，关键决策还是人做
- 修正：DSPy 加速，不是完全自动

## Hermes Agent

### 最简单的解释

Hermes 不完全是框架，更像是一个"智能体平台"——既有模型（Hermes LLM），也有工具调用、记忆、技能等系统。

### 与其他框架的区别

| 维度     | LangChain | LangGraph | OpenAI SDK | Hermes      |
| -------- | --------- | --------- | ---------- | ----------- |
| 定位     | 组件库    | 工作流    | Agent 框架 | 完整平台    |
| 模型     | 支持任何  | 支持任何  | OpenAI     | Hermes 专属 |
| 工具调用 | 基础      | 完善      | 原生       | 高度定制    |
| 学习曲线 | 平缓      | 陡        | 中等       | 中等        |
| 社区     | 很大      | 增长中    | 官方       | 小众        |

### 关键特性

✅ 内置技能系统 - 预定义常用任务
✅ 多轮记忆 - 跨会话保留信息
✅ 自主规划 - Agent 自己决定下一步
✅ 工具调用 - 稳定的结构化输出

### 关键启发

不要只看框架名字，要看：

1. **模型能力** - 是否稳定产生你需要的格式
2. **工具集成** - 是否有现成的连接器
3. **社区支持** - 问题时有人帮吗
4. **学习成本** - 3 天能上手吗
5. **长期维护** - 1 年后还更新吗

## Learning Agent

### 最简单的解释

Learning Agent 指系统能从失败、用户反馈、评估结果中改进。这不一定是重新训练模型，也可以是更新 Prompt、调整 Retriever、补充规则。

### 常见的学习方式

| 方式            | 例子                                  | 风险           |
| --------------- | ------------------------------------- | -------------- |
| **Prompt 优化** | 用户反馈"太啰嗦" → 加上"简洁回答"     | 低             |
| **规则添加**    | 发现某类问题经常出错 → 加特殊处理     | 低             |
| **数据补充**    | 发现知识库缺少某个话题 → 补充相关文档 | 低             |
| **模型微调**    | 用线上数据微调模型                    | 高（可能污染） |

### 常见陷阱

❌ **陷阱 1**：线上反馈直接变行为

```
流程（错误）：
1. 用户说"你这个答案不对"
2. 系统立即更新 Prompt
3. 结果：被恶意用户利用，系统被污染

流程（正确）：
1. 记录失败样本
2. 人工检查、标注原因
3. 加入测试集，验证改进不会破坏其他情况
4. 审批通过后上线
```

❌ **陷阱 2**：学习覆盖面太广

```
错误：系统对所有问题都快速学习，改得太频繁
正确：只学习高频问题，低频问题保持稳定
```

❌ **陷阱 3**：没有回滚机制

```
错误：改了 Prompt，发现更差，但没办法快速回滚
正确：每次改变都版本化，能快速回滚
```

# 评估

Evaluation 是把“感觉效果不错”变成“系统可持续改进”的方法。没有 eval，prompt、模型、RAG、Agent 和工具调用的变化都只能靠主观试用判断，迟早会被回归问题拖住。

用一套标准，量化打分 / 判定好坏，检验大模型、提示词、智能体、API 输出结果够不够准、稳、合规、好用。

## 核心目的

- 测出模型回答正确率
- 对比多个模型谁更强
- 排查幻觉、胡说、违规、答非所问
- 验收 Prompt、Agent 流程是否合格
- 上线前把关、上线后监控质量

## 三个层次的评估

**Level 1：人工试用** - 自己试用、问几个朋友，快但不完整

**Level 2：标准数据集** - 用 100+ 测试用例批量跑分，客观但耗时

**Level 3：线上监控** - 上线后持续收集真实反馈，发现真实问题但需要基础设施

## 评估指标表

| 指标       | 说明                  | 目标      |
| ---------- | --------------------- | --------- |
| 准确率     | 答案是否正确          | ≥ 90%     |
| 相关性     | 有没有跑偏            | ≥ 95%     |
| 事实一致性 | 有无编造（防幻觉）    | ≥ 99%     |
| 流畅度     | 语句通顺              | ≥ 85%     |
| 合规性     | 是否涉黄暴敏感        | 100%      |
| 召回率     | 有没有搜到内容（RAG） | ≥ 80%     |
| 延迟       | 响应时间              | ≤ 500ms   |
| 成本       | token 消耗            | ≤ 1000/次 |

## 常见误解

❌ **误解 1**：有人工测试就够了

- 人工样本太少、不系统
- 修正：必须有自动化、量化评估

❌ **误解 2**：准确率 95% 就一定安全

- 准确率只是一个指标，还要看幻觉率、成本、延迟
- 修正：多维度评估

❌ **误解 3**：评估是开发完成后的事

- 应该从需求阶段定义评估标准
- 修正：评估贯穿全流程

## 实战：如何建立评估体系

**第 1 步：定义成功标准**

不好的标准："感觉准确"
好的标准："准确率 ≥ 90%、幻觉率 < 1%、响应时间 < 1s"

**第 2 步：准备测试集**

收集 100+ 代表性问题：简单 10%、中等 60%、复杂 30%
包含边界情况：模棱两可、多步推理、容易误导

**第 3 步：定义打分规则**

定性评估（人工）：1-5 分
定量评估（自动）：准确率 = 正确 / 总数

**第 4 步：执行并记录**

制作表格追踪：日期、模型、准确率、幻觉率、延迟、成本、是否通过

**第 5 步：根据结果改进**

准确率不足 → 优化 Prompt / 换模型
幻觉率高 → 告诉模型"不知道就说不知道" / 用 RAG
成本高 → 缩短上下文 / 用更小模型
